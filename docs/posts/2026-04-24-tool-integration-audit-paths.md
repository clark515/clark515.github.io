# 从标准 MCP 到平台封装 API：智能体工具接入的三类形态与审计路径

上一篇《MCP 与智能体平台安全初探》主要想澄清一件事：客户现场的“工具接入”并不天然等于标准 MCP。很多平台虽然对外也说“支持 MCP”或“支持工具调用”，但真正落地时，底层实现可能是完全不同的几套东西。

这也是为什么第二篇不继续展开协议细节，而是回到更实际的问题：

**面对一个支持工具接入的智能体平台，应该先判断它属于哪一类目标，再决定从哪里开始审。**

这看起来像方法问题，实际上直接影响结果质量。很多审计做不深，不是因为 payload 不够多，也不是因为测试手法不够激进，而是第一步就把目标看错了。

## 一、一个常见误区：把所有工具接入都当成标准 MCP

MCP 这两年很热，很多厂商也开始对外宣传“兼容 MCP”“支持 MCP 接入”或“提供 MCP 工具能力”。但这句话在技术上，可能对应三种完全不同的形态：

- 真正按 MCP 规范暴露能力的标准 MCP server
- 把普通 HTTP API 包装成 tool 的平台抽象层
- 统一管理 server、凭据、策略和审计的 MCP 管理平台

这三类东西可以同时出现在一个系统里，但安全重点并不一样。

如果不做区分，通常会出现两种偏差：

- 要么只盯协议字段，漏掉后端真正高风险的接口和控制面
- 要么直接按传统 Web/API 去测，忽略了模型调用链和元数据进入上下文后的风险放大

所以在我看来，审计智能体工具接入时，第一步不是枚举协议，而是先做目标识别。

## 二、三类形态，三种不同的审计起手式

先把差异压成一句话：

| 类型 | 重点看什么 | 容易漏什么 |
|------|------------|------------|
| 标准 MCP server | 协议暴露面、模型可调用面 | 元数据注入、能力边界失控 |
| HTTP API 工具封装 | tool 到后端接口的真实映射 | 越权、SSRF、批量误触发、敏感返回值 |
| MCP 管理平台 | 注册、租户、凭据、策略、审计这些控制面 | 平台级绕过、跨租户影响、假审计 |

### 1. 标准 MCP server：先看协议暴露面和模型可调用面

如果目标本身就是标准 MCP server，通常会有比较明显的协议特征：

- 有明确的 `tools`、`resources`、`prompts`
- 有相对完整的 schema 和元数据
- 会话、能力暴露和调用流程比较独立

这类系统最容易给人一种“结构很清晰，所以风险也清晰”的错觉。但从安全角度看，真正关键的是：

**模型最终能在什么条件下调用这些能力。**

因此这类目标的第一轮审计，我更倾向于按这个顺序走：

1. 拉全能力清单
2. 按 `read / write / exec / broker` 做危险度分级
3. 检查 description、示例、schema 是否会误导模型
4. 看高危动作有没有确认、隔离和日志

这类目标里，重点通常集中在四个方向：

- 工具元数据注入
- 高危能力暴露过宽
- resource 暴露超范围
- tool output 导致后续 action 污染

它们的问题核心，不是“协议字段写错了”，而是“模型驱动下的调用边界失控了”。

### 2. 平台封装的 HTTP API 工具：先找真实后端，而不是先看 tool 名字

第二类在客户环境里往往更常见。平台对外展示的是 tool，但真正执行的仍然是传统 HTTP API。

典型架构通常是：

1. 平台维护一组内部接口
2. 平台给这些接口配置名称、描述、参数和调用模板
3. Agent 侧看到的是统一的工具层

这时如果只盯着 `tool description` 和 `input schema`，往往只能看到抽象层，真正的问题藏在后面：

- tool 最终打到哪个接口
- 用的是谁的身份
- 参数是怎么被拼进去的
- 返回值会不会把敏感信息带回来

所以这类目标的审计起手式应该变成：

1. 找 tool 到接口的映射
2. 找接口的鉴权主体和租户边界
3. 看参数拼接、模板渲染、动态路径和二次请求
4. 再判断模型输入能否把这些点打穿

这类系统里，更值得优先怀疑的通常不是“协议漏洞”，而是：

- 越权调用内部管理接口
- 可控 URL 或 path 触发 SSRF
- 批量操作、删除、变更类接口被无约束暴露成工具
- 返回值泄露 token、内部主键、调试堆栈或策略信息

换句话说，这类系统的重点不是“tool 看起来安不安全”，而是“被包装成 tool 的后端接口本来安不安全”。

### 3. MCP 管理平台：先看控制面，再看执行面

第三类更像 MCP 的控制平面。它不一定直接执行所有 tool，但它负责：

- server 的注册、审核、发布和下架
- 插件或工具市场
- 凭据托管
- 路由分发
- 权限策略
- 调用审计和审批

这类平台的风险更集中，因为一旦这里出问题，影响的就不是一个 tool，而是一批 tool、一批 server，甚至一个租户域里的整条调用链。

所以它的审计入口和前两类不同。第一轮更值得优先看的是：

1. 注册和审核流程能否被绕过
2. 多租户数据和凭据是否严格隔离
3. UI 上的策略是否真实落到调用链
4. 审批、审计、回滚是否只是“展示能力”

这类系统里最有价值的问题，未必会表现成传统意义上的高危 RCE，但很可能是更关键的平台级缺陷：

- 恶意 server 混入
- 凭据跨租户复用
- 策略绕过
- 审计绕过

如果只把它当成“再复杂一点的 MCP server”去扫，通常会漏掉这些真正关键的点。

## 三、一个更稳的审计顺序

我现在更认可的顺序，不是“先枚举协议，再尝试 payload”，而是下面五步。

### 第一步：先识别形态

先判断目标到底属于哪一类，或者三类里哪几类叠加。

如果这一层判断错了，后面所有规则、验证点和预期漏洞类型都可能跑偏。

### 第二步：再画调用链

至少要画清楚三件事：

- 模型看到了什么
- 平台转发了什么
- 后端真正执行了什么

只有把这三层拆开，才知道安全边界到底在哪里。

### 第三步：做能力分级

无论是哪一类目标，能力分级都值得先做。最简单也最有效的做法，还是按下面四类分：

- `read`
- `write`
- `exec`
- `broker`

这样后面再看确认、审批、日志和隔离，就不会混。

### 第四步：挑重点点位验证

验证不必一开始铺很大。通常先盯几个高价值点就够：

- 高危 tool 的确认和约束
- tool 到后端接口的映射关系
- resource 暴露范围
- 平台控制面的隔离和策略

### 第五步：最后再沉淀检查表

很多团队一上来就写 checklist，但如果前面没有做过类型识别和调用链分析，这份 checklist 往往只会停在泛化描述上。

更稳的做法是先做一轮真实目标分析，再把检查项沉淀下来。

## 四、一个抽象案例

假设某个平台对外宣称“支持 MCP 工具接入”。第一眼看过去，前台已经能看到整齐的 tool 名称、参数 schema 和调用记录，似乎很像标准 MCP。

但继续往下看，发现：

- 所有 tool 实际都映射到平台内部 HTTP API
- 调用身份不是最终用户，而是平台统一托管的服务账号
- 部分高风险接口同时支持查询和修改
- 日志页面只记录前台 tool 名称，不记录真实后端接口

这时候如果只做标准 MCP 枚举，很可能只能得到“schema 正常、工具列表清晰、调用页面有日志”这样的表层结论。

但如果按第二类形态去审，风险面马上就不一样了：

- 统一服务账号是否导致越权
- 高危接口是否被 tool 抽象掩盖
- 返回值是否泄露内部信息
- 前台可见的审计是否只是部分审计

同一个目标，识别路径不同，最后发现的问题层级会完全不同。

## 五、结语

对智能体平台来说，“工具接入”越来越像一个统一概念；但对安全审计来说，不能接受这种过度统一。

名字相同，不代表控制边界相同。  
界面相同，不代表后端实现相同。  
都叫 MCP，不代表都该按 MCP 一套方法去打。

如果只用一句话总结这篇文章，我会这么说：

**审计智能体工具接入时，先判断它是什么，再决定怎么测；先看调用链和控制面，再谈 payload 和漏洞类型。**
